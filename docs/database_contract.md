# Database Contract

## Purpose
This document explains what data currently exists in the Guardian prototype, where it comes from, what structure it follows, and what the database layer should be prepared to store.

The goal is to help the database part start with the correct data model, without needing to guess field names or object structure.

---

## 1. Main data categories

The current Guardian prototype works with 4 main categories of data:

1. telemetry data
2. alert data
3. operator action data
4. validation / metrics data

At the current phase, telemetry and alerts are the most important.

---

## 2. Telemetry data

### What it is
Telemetry is the raw input data received by the Guardian. It represents aircraft or testbed state over time.

### Where it comes from
Current phase:
- replayed CSV scenario files in `data/scenarios/`

Later phase:
- live telemetry from the RC aircraft / sensor node

### Source files
Main reference files:
- `docs/telemetry_schema.md`
- `docs/architecture.md`
- `docs/dashboard_contract.md`

Current replay scenario files:
- `data/scenarios/normal_flight.csv`
- `data/scenarios/packet_loss.csv`
- `data/scenarios/sensor_dropout.csv`
- `data/scenarios/gps_jump.csv`
- `data/scenarios/low_battery.csv`
- `data/scenarios/out_of_order_packets.csv`
- `data/scenarios/duplicate_packet.csv`
- `data/scenarios/frozen_imu.csv`
- `data/scenarios/gps_fix_loss.csv`
- `data/scenarios/gps_imu_inconsistency.csv`
- `data/scenarios/combined_fault.csv`

### What it looks like
Each telemetry row contains fields such as:

- `timestamp_ms`
- `packet_id`
- `node_id`
- `accel_x_g`
- `accel_y_g`
- `accel_z_g`
- `gyro_x_dps`
- `gyro_y_dps`
- `gyro_z_dps`
- `temperature_c`
- `pressure_hpa`
- `altitude_est_m`
- `battery_voltage_v`
- `low_power_flag`
- `gps_lat_deg`
- `gps_lon_deg`
- `gps_alt_m`
- `gps_speed_mps`
- `gps_fix_status`
- `satellite_count`
- `link_status`
- `mode_state`

### What the database should do with it
The database should be ready to store telemetry rows as time-series input data.

Minimum expectation:
- one record per telemetry packet / row
- preserve timestamps and packet IDs
- preserve all sensor and state fields

---

## 3. Alert data

### What it is
Alerts are generated by the Guardian after processing telemetry.

### Where it comes from
Source:
- Guardian processing logic in `guardian/engine.py`
- rule-based detection in `guardian/rules.py`
- alert construction in `guardian/alerts.py`

### Reference files
- `docs/alert_schema.md`
- `docs/dashboard_contract.md`

### What it looks like
Each alert contains fields such as:

- `timestamp_ms`
- `packet_id`
- `node_id`
- `severity`
- `confidence`
- `reason_code`
- `reason_text`
- `recommended_action`
- `alert_status`

### Example reason codes
- `PACKET_LOSS`
- `IMU_DROPOUT`
- `LOW_BATTERY`
- `GPS_JUMP`
- `GPS_IMU_INCONSISTENCY`
- `OUT_OF_ORDER_PACKET`
- `DUPLICATE_PACKET`
- `IMU_FROZEN`
- `GPS_FIX_LOSS`

### What the database should do with it
The database should store each generated alert as a structured record.

Minimum expectation:
- one record per alert
- preserve link to telemetry context through timestamp / packet / node
- preserve reason code, severity, confidence, and recommended action

---

## 4. Operator action data

### What it is
Operator action data represents actions taken from the dashboard or by a user in response to an alert.

### Current phase
This is mostly planned, not fully active yet.

### Expected structure
Suggested fields:

- `timestamp_ms`
- `packet_id`
- `node_id`
- `reason_code`
- `action_type`
- `operator_note`

### Expected action types
- `acknowledge`
- `override`
- `escalate`
- `request_verification`
- `resolve`

### What the database should do with it
The database should be ready to store operator actions later as separate records linked to relevant alerts.

---

## 5. Validation / metrics data

### What it is
This data is generated during replay-based validation and helps measure system behavior.

### Where it comes from
Generated artifacts:
- `results/metrics/scenario_metrics.csv`
- `results/metrics/expected_vs_observed.csv`
- `results/metrics/validation_summary.md`

### What it looks like
Examples include:
- scenario name
- rows processed
- alerts generated
- warning alerts
- critical alerts
- observed reason codes
- expected reason codes
- pass/fail validation status

### What the database should do with it
Optional for now.
Later, the database may store validation runs and scenario-level metrics for analysis.

---

## 6. Minimum database scope for Phase 1

To start simply, the database part should focus on these 2 core entities:

### A. Telemetry
Store:
- all telemetry fields from scenario/live input rows

### B. Alerts
Store:
- all alert fields generated by the Guardian

That is enough to support the current Guardian prototype and prepare for dashboard integration.

---

## 7. Suggested next database scope for Phase 2

After telemetry and alerts are working, expand to:

### C. Operator actions
Store user responses to alerts

### D. Validation results
Store scenario metrics and pass/fail validation outputs

---

## 8. Field consistency requirement

Field names should remain consistent with:
- `docs/telemetry_schema.md`
- `docs/alert_schema.md`
- `docs/dashboard_contract.md`

The Guardian currently uses those names directly, so the database should align with them.

---

## 10. Practical expectation

The database layer does not need to implement final production architecture at this stage.
It only needs to create a clean structure that can store:
- telemetry input
- Guardian alert output
- later operator actions
- optionally validation artifacts
