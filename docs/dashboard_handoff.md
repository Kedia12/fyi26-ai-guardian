# Dashboard Handoff

## Purpose
This document explains what data the dashboard part should expect from the Guardian prototype, what should be displayed first, and what future operator interactions may be supported.

The goal is to help the dashboard part start with the correct structure and priorities without needing to guess field names or alert meanings.

---

## 1. Main dashboard inputs

The dashboard will mainly work with 2 data categories:

1. telemetry data
2. alert data

Later, it may also support:
3. operator action data

---

## 2. Telemetry data

### What it is
Telemetry is the live or replayed aircraft/testbed state processed by the Guardian.

### Where it comes from
Current phase:
- replay scenario files in `data/scenarios/`

Later phase:
- live telemetry from the RC aircraft sensor node

### Main reference files
- `docs/telemetry_schema.md`
- `docs/dashboard_contract.md`
- `docs/architecture.md`

### Main fields the dashboard may display
- `timestamp_ms`
- `packet_id`
- `battery_voltage_v`
- `temperature_c`
- `pressure_hpa`
- `altitude_est_m`
- `gps_speed_mps`
- `gps_fix_status`
- `satellite_count`
- `link_status`
- `mode_state`

### Optional motion fields for advanced visualization
- `accel_x_g`
- `accel_y_g`
- `accel_z_g`
- `gyro_x_dps`
- `gyro_y_dps`
- `gyro_z_dps`

---

## 3. Alert data

### What it is
Alerts are structured outputs generated by the Guardian after detecting suspicious anomalies.

### Where it comes from
- `guardian/rules.py`
- `guardian/engine.py`
- `guardian/alerts.py`

### Main reference files
- `docs/alert_schema.md`
- `docs/dashboard_contract.md`

### Alert fields to display
- `timestamp_ms`
- `packet_id`
- `node_id`
- `severity`
- `confidence`
- `reason_code`
- `reason_text`
- `recommended_action`
- `alert_status`

### Example reason codes
- `PACKET_LOSS`
- `IMU_DROPOUT`
- `LOW_BATTERY`
- `GPS_JUMP`
- `GPS_IMU_INCONSISTENCY`
- `OUT_OF_ORDER_PACKET`
- `DUPLICATE_PACKET`
- `IMU_FROZEN`
- `GPS_FIX_LOSS`

---

## 4. Minimum dashboard scope for Phase 1

The first useful version of the dashboard should focus on:

### A. Live telemetry summary
Display:
- packet / timestamp
- battery voltage
- GPS speed
- altitude estimate
- GPS fix status
- link status
- mode state

### B. Alert panel
Display:
- severity
- reason code
- reason text
- confidence
- recommended action
- alert status

### C. Alert history / incident list
Display:
- alerts over time
- packet/timestamp reference
- list of triggered anomalies

This is enough for the first prototype demonstration.

---

## 5. Recommended dashboard priorities

### First priority
Show alerts clearly and make them easy to interpret.

### Second priority
Show the key telemetry values that help explain the alert context.

### Third priority
Show a simple incident timeline or alert history.

The dashboard does not need to display every telemetry field in the first version.

---

## 6. Operator actions

### Current phase
Operator actions are mostly planned, not required for the first basic dashboard version.

### Later possible actions
- `acknowledge`
- `override`
- `escalate`
- `request_verification`
- `resolve`

### Suggested operator action fields
- `timestamp_ms`
- `packet_id`
- `node_id`
- `reason_code`
- `action_type`
- `operator_note`
